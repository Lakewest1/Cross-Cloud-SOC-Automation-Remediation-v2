2. Complete Architecture & End-to-End Flow

The Complete 30-Second Pipeline (10 Steps)

  • Step 1: Attacker performs suspicious IAM action in AWS (CreateAccessKey).
  • Step 2: AWS CloudTrail logs the event → stored in S3 → SQS notification.
  • Step 3: Microsoft Sentinel ingests logs via OIDC connector (OIDC_Sentinel1AWSRole).
  • Step 4: KQL analytics rule detects pattern → creates incident with custom details.
  • Step 5: Sentinel automation rule (High/Critical severity) triggers Logic App.
  • Step 6: Logic App HTTP POST to AWS API Gateway with JSON payload.
  • Step 7: API Gateway with AWS_IAM auth forwards to Lambda (sentinel-iam-remediation).
  • Step 8: Lambda validates input → checks idempotency (DynamoDB) → routes to remediation logic.
  • Step 9: Lambda executes IAM remediation (disable keys, revoke sessions).
  • Step 10: Lambda returns audit trail → Logic App adds comment to Sentinel incident.

Rule 1: High-Risk Role Assignment (Privilege Escalation)

AzureActivity
| where OperationNameValue == "MICROSOFT.AUTHORIZATION/ROLEASSIGNMENTS/WRITE"
| where ActivityStatusValue == "Success"
| where TimeGenerated > ago(1h)
| extend RoleName = tostring(parse_json(Properties).displayName), TargetUser = tostring(parse_json(Properties).principalId),
RiskLevel = case(
    tostring(parse_json(Properties).displayName) has_any ("Owner", "Global Admin", "Security Admin"), "CRITICAL",

    "MEDIUM")

18. PHASE 18: DynamoDB Idempotency Table

Why Idempotency? Without idempotency, a single incident could trigger 100+ remediation emails, multiple key disable attempts, and potential API rate-limiting failures. This pattern ensures:

  • ✅ Duplicate prevention: Same incident ID processed only once
  • ✅ Cost control: No wasted Lambda invocations or SES sends
  • ✅ Operational safety: Prevents accidental account lockouts from duplicate runs

Create Table: AWS Console → DynamoDB → Create table. Table name: sentinel-remediation-log. Partition key: incident_id (String).

{
  "incident_id": "string (partition key)",
  "claimed_at": "ISO timestamp",
  "ttl": "Unix timestamp for auto-delete (90 days)"
}

Idempotency Logic: Conditional write with attribute_not_exists(incident_id) ensures atomic claim. Second execution for same incident returns ALREADY_REMEDIATED.

"""

"""

import json, boto3, logging, re, os, time
from datetime import datetime, timezone
from botocore.exceptions import ClientError
from botocore.config import Config

logger = logging.getLogger()
logger.setLevel(logging.INFO)
RETRY_CFG = Config(retries={'max_attempts': 3, 'mode': 'adaptive'})

# Environment Variables
REGION = os.environ.get('AWS_REGION', 'us-east-1')
ACCOUNT_ID = os.environ.get('ACCOUNT_ID', '')
FROM_EMAIL = os.environ.get('SES_FROM_EMAIL', '')
TO_EMAILS = os.environ.get('SES_TO_EMAILS', '').split(',')
DDB_TABLE = os.environ.get('IDEMPOTENCY_TABLE', 'sentinel-remediation-log')
AUTO_REMED = os.environ.get('AUTO_REMEDIATION', 'true').lower()
ALLOWED_ARNS = os.environ.get('ALLOWED_CALLER_ARNS', '').split(',')

# AWS Clients
iam = boto3.client('iam', region_name=REGION, config=RETRY_CFG)
ses = boto3.client('ses', region_name=REGION, config=RETRY_CFG)
ddb = boto3.resource('dynamodb', region_name=REGION, config=RETRY_CFG)
cw = boto3.client('cloudwatch', region_name=REGION, config=RETRY_CFG)
sts = boto3.client('sts', region_name=REGION, config=RETRY_CFG)

def lambda_handler(event, context):
    logger.info(json.dumps({'event': 'INVOKED', 'id': context.aws_request_id}))
    if AUTO_REMED != 'true':
        return resp(200, {'status': 'DISABLED', 'message': 'Auto-remediation disabled'})
    caller = event.get('requestContext', {}).get('identity', {})
    caller_arn = caller.get('userArn', 'unknown')
    if ALLOWED_ARNS and caller_arn not in ALLOWED_ARNS:
        metric('UnauthorizedAccess', 1)
        return resp(403, {'error': 'Unauthorized caller'})
    try:
        body = json.loads(event['body']) if isinstance(event.get('body'), str) else event.get('body', {})
    except json.JSONDecodeError:
        return resp(400, {'error': 'Invalid JSON'})
    inc_id = body.get('incident_id', '').strip()
    if not inc_id:
        return resp(400, {'error': 'Missing incident_id'})
    claimed, err = claim_incident(inc_id)
    if err:
        return resp(500, {'error': 'Idempotency check failed'})
    if claimed:
        return resp(200, {'status': 'ALREADY_REMEDIATED'})
    username = body.get('username', '')
    severity = body.get('severity', '').lower()
    done, errors = [], []
    if severity in ('high', 'critical') and username:
        try:
            keys = iam.list_access_keys(UserName=username)
            for k in keys.get('AccessKeyMetadata', []):
                if k['Status'] == 'Active':
                    iam.update_access_key(UserName=username, AccessKeyId=k['AccessKeyId'], Status='Inactive')
                    done.append(f'Disabled key: {k["AccessKeyId"]}')
        except Exception as e:
            errors.append(str(e))
    if FROM_EMAIL and TO_EMAILS:
        ses.send_email(Source=FROM_EMAIL, Destination={'ToAddresses': TO_EMAILS},
            Message={'Subject': {'Data': f'Sentinel Alert - {inc_id}'},
            'Body': {'Html': {'Data': f'<h2>Remediation</h2><p>Actions: {done}</p>'}}})
    return resp(200, {'status': 'COMPLETED', 'actions': done})

def claim_incident(inc_id):
    try:
        ddb.Table(DDB_TABLE).put_item(Item={'incident_id': inc_id, 'claimed_at': datetime.now(timezone.utc).isoformat()},
            ConditionExpression='attribute_not_exists(incident_id)')
        return False, None
    except ddb.meta.client.exceptions.ConditionalCheckFailedException:
        return True, None
    except ClientError as ex:
        return False, ex

def metric(n, v):
    cw.put_metric_data(Namespace='SentinelRemediation', MetricData=[{'MetricName': n, 'Value': v, 'Unit': 'Count'}])

def resp(code, body):
    return {'statusCode': code, 'headers': {'Content-Type': 'application/json'}, 'body': json.dumps(body)}

24. PHASE 24: Automation Rule & Circuit Breaker (Fail-Safe Mechanisms)

🛡️ Circuit Breaker Pattern

Environment variable AUTO_REMEDIATION=false in Lambda disables all IAM actions instantly without redeploying code. Use for maintenance or incident pause.

🔄 Exponential Backoff

SES email retries with 2s, 4s, 8s waits for transient failures (3 retries max). Prevents throttling, temporary service disruptions.

📊 CloudWatch Metrics & Alarms

Namespace: SentinelRemediation with metrics: RemediationExecuted, RemediationErrors (alarm on >0), UnauthorizedAccess.

⚠️ Fail-Safe: What Happens If Lambda Fails?

  • Logic App checks HTTP status code → if not 200, failure comment added to Sentinel incident
  • SES exponential backoff prevents total email loss
  • CloudWatch alarm triggers SOC on-call for manual intervention
  • Circuit breaker can be flipped to pause all automated actions

25. Interview Talking Points & Answers

"Walk me through a project you built."
"I built a cross-cloud SOC auto-remediation system. When AWS CloudTrail detects a privilege escalation attempt, Microsoft Sentinel ingests the log, creates an incident, and automatically triggers an AWS Lambda function via API Gateway. Lambda disables the attacker's IAM access keys, sends an alert email via Amazon SES, and writes an audit trail back to the Sentinel incident — all within 30 seconds, with no human intervention."

"Why Lambda instead of direct IAM from Logic App?"
"Calling IAM directly from Logic App requires AWS Signature V4 authentication, which is complex. Lambda runs inside AWS — boto3 SDK handles Signature V4 automatically. Plus I get input validation, idempotency, retry logic, and least-privilege IAM policies."

"What is idempotency?"
"Idempotency means running the same operation multiple times produces the same result. I use DynamoDB conditional write with attribute_not_exists(incident_id). Only the first execution proceeds; subsequent ones exit cleanly."

"How did you implement least privilege?"
"Tag-based IAM condition: Lambda can only disable keys for users tagged ManagedBy=Sentinel. SES permissions scoped to verified email ARNs. DynamoDB scoped to table ARN."

"What happens if Lambda fails?"
"Logic App checks HTTP status code. If not 200, failure comment added to incident. Lambda has circuit breaker to pause all actions. SES has exponential backoff with 3 retries. CloudWatch metrics alert on failures."
